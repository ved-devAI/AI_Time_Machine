## What is already winner-quality

### 1. Evidence validation is a real differentiator

The application does not merely show citations. It rejects unknown event IDs,
wrong commits, and files borrowed from a different event. Artifact digests bind
the GPT result to the exact OrbitCart evidence snapshot. This is much stronger
than decorative source links.

### 2. The project treats uncertainty honestly

The flagship conclusion is labeled inferred, not confirmed. The UI separately
shows what Git can prove, what it cannot prove, and what risk remains. That is a
credible answer to hallucination concerns in developer tooling.

### 3. The stale-price story is memorable

The performance-pressure to unsafe-cache to incident to rollback to safe-fix
sequence is understandable without repository-specific knowledge. The Bug
Origin Trace gives the project a recognizable visual moment.

### 4. The zero-key path is reliable

Judges can run the project without buying API credits or configuring secrets.
The application labels the hosted data as a verified Git snapshot and the
Codex results as validated build-time artifacts rather than pretending a live
model call occurred.

### 5. The repository shows engineering discipline

The verification command is short, deterministic, CI-backed, and exercises the
judge-facing workflow. The README explains artifact reproduction and the local
generic-repository workflow.

## Where a skeptical judge will deduct points

### P0 — The submission package is incomplete

The YouTube demo, Devpost copy, testing instructions, final link verification,
and required `/feedback` session ID are still pending. A technically excellent
project can lose entirely because the judge cannot understand or reproduce it
within the review window.

### P1 — The public AI experience can look canned

OrbitCart Ask the Repo exposes exactly three questions. Their GPT-5.6 Sol
answers were generated through Codex, committed, validated, and replayed. This
is an intentional reliability design, but without a concise explanation a
judge can interpret it as three prewritten responses attached to a synthetic
demo.

The winning framing is:

> GPT proposes a causal interpretation once. AI Time Machine binds that result
> to the repository evidence and refuses to display it when any event, commit,
> file, causal stage, or evidence digest is invalid.

Do not frame the hosted experience as a live repository chatbot.

### P1 — OrbitCart is deliberately idealized

OrbitCart is a genuine Git repository, but its history was authored to contain
structured reasons, risks, incidents, rollback, resolution, and verification.
It proves that the pipeline reads Git rather than hard-coded timeline cards. It
does not by itself prove usefulness on messy repository history.

The demo should therefore include a short proof using AI Time Machine's own
history or another small real repository that was not created for the product.
The most persuasive moment is not a perfect answer; it is the tool explicitly
saying that rationale or production evidence is missing.

### P1 — The evidence coverage metric overstates completeness

The UI currently calculates evidence coverage as confirmed commit events divided
by all commit events. For a generic repository, commit existence and diffs may
be confirmed while the reason and risks remain `not recorded`. The interface can
therefore show `100%` coverage even when engineering rationale coverage is near
zero.

Replace the single percentage with clearly named measures, for example:

- Commit and diff evidence: 100%
- Recorded rationale: 0%
- Recorded risk: 0%
- Incident or fix classification: 0%

At minimum, rename the existing measure to **Commit evidence verified** so it
does not imply complete historical understanding.

### P1 — Branch missing-evidence reporting is incomplete

The branch-context engine currently populates `missing_evidence` only when no
changed files exist. A range can contain many commits with absent rationale,
absent risks, and no incident links while reporting no missing evidence.

The report should explicitly identify:

- range commits whose rationale is not recorded;
- changed files with no prior incident or fix evidence;
- absence of recorded risks;
- absence of deployment, issue-tracker, review, or production telemetry evidence.

This repair is closely aligned with the product contract and is safer than
adding a new integration during M5.

### P2 — The 100% grounding score is internally defined

The 15/15 score is strong regression evidence. It proves that the committed
artifacts continue to match the expected OrbitCart causal chain and that known
invalid cases are rejected. It is not an independent accuracy benchmark because
the expected results and the demo history are maintained by the project.

Devpost language should call it a **deterministic grounding regression
scorecard**, not a generalized 100% causal-accuracy result.

### P2 — The trace is visually dense

The six-stage trace is the best visual in the project, but six equal cards plus
the verdict, unknowns, and risks compete for attention. In the video, guide the
viewer through four beats first: pressure, likely origin, observed incident,
verified fix. Let containment and verification support the story rather than
requiring the viewer to read every card.

### P2 — The handoff contains stale self-analysis counts

The previous handoff recorded nine timeline events, 43 changed files, and eight
range commits. The July 20 review produced 11 events, 49 changed files, and 10
range commits. Run-specific counts should be refreshed after final code changes
or omitted from durable marketing copy.

## Suggested final positioning

### One-line pitch

AI Time Machine reconstructs why a codebase changed, then checks every AI
citation against the exact Git events, commits, and files before displaying it.

### Differentiator

Most repository assistants generate plausible explanations. AI Time Machine
treats those explanations as untrusted until their causal stages and citations
match observable repository references, while clearly separating reference
integrity from semantic proof.

### Honest OpenAI statement

GPT-5.6 Sol in Codex materially supported architecture, implementation, review,
evaluation, and generation of the committed causal-analysis artifacts. The
public runtime replays those validated artifacts without claiming a live model
call or requiring the judge to provide an API key.
